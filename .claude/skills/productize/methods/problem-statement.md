# Method — Problem Statement (PRD 1.1)

*Re-authored from Dean Peters' Product-Manager-Skills (`problem-statement`, `press-release`), CC BY-NC-SA 4.0 — attribution per Build Stance.*

A sharp problem statement names **who** has the problem, **what** it is, **when/where** it bites, and **why** existing options fall short — *before* any solution. Pair it with a working-backwards headline: if you can't write a press-release headline a target customer would be excited by, the problem isn't sharp yet.

## Apply to PRD 1.1 (Project Overview)
- **Who** + context — one sentence.
- **Problem** — the pain, with frequency/severity. No solution language.
- **Proposed solution** — one paragraph; the smallest thing that addresses the problem.
- **Headline test** — a press-release headline a target customer would click; if it's vague, sharpen the problem, not the prose.

**Anti-pattern:** stating the solution as the problem ("they need our app"). The problem exists with or without you.
